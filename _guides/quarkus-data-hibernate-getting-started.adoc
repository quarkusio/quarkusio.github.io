== Overview

Database access is something almost every application needs, whether it's persisting data or querying tables.
In this guide we start from scratch and build up a repository for an online bookstore, going from your first SQL query to compile-time validated repositories and reactive database access. By the end you'll have a solid understanding of how data access works in Quarkus.

== Introduction

You're working on a physical bookstore website, in which you want users to be able to browse books with titles.
In this tutorial, we won't deal with the frontend and the user interface, we just want to connect to the database.

We'll start this tutorial by creating a new empty Quarkus application:

:create-app-artifact-id: quarkus-data-tutorial
:create-app-platform-version: 3.37.0.CR1
include::{includes}/devtools/create-app.adoc[]

Let's start by importing the extensions:

:add-extension-extensions: quarkus-data-hibernate,quarkus-jdbc-postgresql
include::{includes}/devtools/extension-add.adoc[]

The first dependency, `quarkus-data-hibernate`, is the Quarkus extension that provides everything you need for database access.
The second dependency, `quarkus-jdbc-postgresql`, is the JDBC driver that tells Quarkus how to talk to a PostgreSQL database. If you're using a different database (MySQL, MariaDB, H2, etc.), pick the corresponding `quarkus-jdbc-*` driver instead. Quarkus Data Hibernate cannot work without a JDBC driver.

That's all you need to start querying your database.

Quarkus Data Hibernate uses the Hibernate annotation processor, a key component that reads your code at compile time, validates queries, and generates implementation classes under `target/generated-sources/annotations/`.
These generated classes are compiled together with your code and wired up automatically by Quarkus. They are also simple to read, in case you want to understand how it works internally.

To enable it, find the `maven-compiler-plugin` in your `pom.xml` (it was generated by `quarkus create app`) and add the annotation processor configuration to it:

[source,xml,role="primary asciidoc-tabs-target-sync-cli asciidoc-tabs-target-sync-maven"]
.pom.xml
----
<plugin>
    <artifactId>maven-compiler-plugin</artifactId>
    <version>${compiler-plugin.version}</version>
    <configuration>
        <parameters>true</parameters> <!-- already present -->
        <annotationProcessorPathsUseDepMgmt>true</annotationProcessorPathsUseDepMgmt>
        <annotationProcessorPaths>
            <path>
                <groupId>org.hibernate.orm</groupId>
                <artifactId>hibernate-processor</artifactId>
            </path>
        </annotationProcessorPaths>
    </configuration>
</plugin>
----

Now run Quarkus continuous testing in a separate tab of your terminal:

[source,bash]
----
quarkus test
----

You don't need to restart Quarkus testing, it will re-run tests continuously after every change.

This SQL script creates sequences for ID generation, the `Publisher` and `Book` tables, and inserts sample data.
Each book is linked to a publisher.
The sequences start past the imported data so that new IDs don't collide.
While developing, Quarkus imports this SQL file automatically at startup, so in this tutorial we have something to start with.

Quarkus Data Hibernate can handle the database schema for you.
Since our `import.sql` already handles the full schema creation, let's tell Hibernate not to manage it and tell Dev Services to run our script when the database container starts.
Add this to `src/main/resources/application.properties`:

[source,properties]
----
quarkus.hibernate-orm.schema-management.strategy=none
quarkus.datasource.devservices.init-script-path=import.sql
----

TIP: JDBC connections are called "Data Sources" in Quarkus. In dev mode, if you don't configure a datasource, Quarkus will automatically start a PostgreSQL container for you via Dev Services. This means you can run `quarkus dev` and start coding without installing or configuring a database. See the xref:datasource.adoc[Datasource guide] for more details.

== Simple SQL query

The simplest thing you can do to access your database is to write native SQL queries directly. Create a file called `BookQueries.java` in the package `org.acme`:

[source,java]
----
package org.acme;

import org.hibernate.annotations.processing.SQL;

import java.math.BigDecimal;
import java.util.List;

public interface BookQueries { // <1>

    record BookRecord(Long id, String title, String isbn, BigDecimal price) {} // <2>

    @SQL("SELECT id, title, isbn, price FROM Book WHERE title = :title") // <3> <4>
    List<BookRecord> findByTitle(String title); // <5>

    @SQL("SELECT id, title, isbn, price FROM Book WHERE proce < :maxPrice")
    List<BookRecord> cheaperThan(BigDecimal maxPrice);
}
----
<1> A repository interface that groups all your query methods for books. The annotation processor generates the implementation for you.
<2> A Java record that maps to the columns returned by the SQL query. Each field name must match a column name in the result set.
<3> `@SQL` sends raw SQL to the database. The `:title` syntax is a named parameter, it avoids SQL injection attacks.
<4> `@SQL` is a Hibernate-specific annotation. Jakarta Persistence 4.0 will introduce a standard https://github.com/jakartaee/persistence/pull/992[`@NativeQuery`] equivalent.
<5> The method signature defines the parameters. The annotation processor generates the implementation.

If SQL is enough for you, this is a perfectly valid use case of `quarkus-data-hibernate`.

We can now create a test in `src/test/java/org/acme/BookQueriesTest.java` that will check everything is working correctly:

[source,java]
----
package org.acme;

import io.quarkus.test.junit.QuarkusTest;
import jakarta.inject.Inject;
import org.junit.jupiter.api.Test;

import java.util.List;

import static org.junit.jupiter.api.Assertions.*;

@QuarkusTest // <1>
public class BookQueriesTest {

    @Inject
    BookQueries bookQueries; // <2>

    @Test
    void shouldFindBookByTitle() {
        List<BookQueries.BookRecord> books = bookQueries.findByTitle("Effective Java"); // <3>
        assertEquals(1, books.size());
        assertEquals("Effective Java", books.getFirst().title());
    }

    @Test
    void shouldReturnEmptyListForUnknownTitle() {
        List<BookQueries.BookRecord> books = bookQueries.findByTitle("Unknown Book");
        assertTrue(books.isEmpty());
    }
}
----
<1> `@QuarkusTest` starts the full Quarkus application for the test, including the database and Dev Services. The data from `import.sql` is loaded automatically.
<2> We inject our `BookQueries` just like we would in application code.
<3> We query for a book we know exists in `import.sql` and verify it's returned.

If you have `quarkus test` running in another terminal, this test will run automatically as soon as you save the file.

Even though raw SQL is a perfectly valid approach, it has a few problems.
First of all, type safety: nobody is going to validate your queries at compile time.

Let's add a test for `cheaperThan` to our `BookQueriesTest`:

[source,java]
----
@Test
void shouldFindCheaperBooks() {
    List<BookQueries.BookRecord> books = bookQueries.cheaperThan(BigDecimal.valueOf(45));
    assertEquals(4, books.size()); // Clean Code (40), Head First Design Patterns (38), The Pragmatic Programmer (42), Kotlin in Action (44)
}
----

The console in `quarkus test` will show

[source,shell]
----
org.hibernate.exception.SQLGrammarException: JDBC exception executing SQL [ERROR: column "proce" does not exist

Hint: Perhaps you meant to reference the column "book.price".

Position: 26] [

SELECT id, title, isbn, price FROM Book WHERE proce < ?
----

This test will fail at runtime with a SQL error because the column `proce` doesn't exist.
Did you initially spot the typo in `cheaperThan`? The column is called `price`, not `proce`, but the compiler won't tell you.
You're probably going to have integration tests to catch that, but it would be faster if the compiler could validate it earlier.

You can fix the test by modifying the `BookQueries.java` file:

[source,java]
----
    @SQL("SELECT id, title, isbn, price FROM Book WHERE price < :maxPrice") // <1>
    List<BookRecord> cheaperThan(BigDecimal maxPrice);
----
<1> Change `proce` to `price` here

and then all the tests will now pass.

Writing SQL by hand makes sense for very simple scenarios, or when you need database-specific features such as window functions, recursive functions and similar advanced patterns.

== Entities and repositories

We can have a faster development model with the help of the compiler and avoid all these problems by mapping our table to a Java class and defining a repository for it.
We now introduce the concept of an Entity: a Java class that maps to a database table.

Create a file called `Book.java` in the package `org.acme`:

[source,java]
----
package org.acme;

import io.quarkus.hibernate.panache.PanacheRepository;
import io.quarkus.hibernate.panache.WithId;
import jakarta.data.repository.Find;
import jakarta.data.repository.Query;
import jakarta.persistence.Entity;

import java.util.List;

@Entity // <1>
public class Book extends WithId.AutoLong { // <2>
    public String title; // <3>
    public String isbn;
    public BigDecimal price;

    public interface Repo extends PanacheRepository<Book> { // <4>
        @Find
        List<Book> findByTitle(String title); // <5>

        @Query("where price < :maxPrice order by title")
        List<Book> cheaperThan(BigDecimal maxPrice); // <6>
    }
}
----
<1> Tells Hibernate that this class maps to a database table. Hibernate will use the class name as the table name and the field names as column names.
<2> Gives the entity a generated `Long` id field automatically.
<3> These public fields map to columns in the `Book` table. Hibernate reads and writes these fields directly.
<4> Defines a repository: a place to put all your query methods for `Book`. By nesting it inside the entity, the queries and the data they operate on live together. You can name this interface whatever you want: `Queries`, `Repo`, `BookQueries`, etc.
<5> `@Find` tells Quarkus Data Hibernate to generate a query based on the method's parameter names. `findByTitle(String title)` becomes a query that matches the `title` field on `Book`. Parameters are inferred from the method signature.
<6> `@Query` lets you write an HQL fragment when you need more control. It avoids the need to write `select b from Book`. The entity type is inferred from the return type `List<Book>`.

Now let's write a test to check that the repository works. Create `src/test/java/org/acme/BookRepositoryTest.java`:

[source,java]
----
package org.acme;

import io.quarkus.test.junit.QuarkusTest;
import jakarta.inject.Inject;
import org.junit.jupiter.api.Test;

import java.util.List;

import static org.junit.jupiter.api.Assertions.*;

@QuarkusTest
public class BookRepositoryTest {

    @Inject
    Book.Repo bookRepo;

    @Test
    void shouldFindBookByTitle() {
        List<Book> books = bookRepo.findByTitle("Effective Java");
        assertEquals(1, books.size());
        assertEquals("Effective Java", books.getFirst().title);
    }

    @Test
    void shouldFindCheaperBooks() {
        List<Book> books = bookRepo.cheaperThan(BigDecimal.valueOf(45));
        assertEquals(4, books.size()); // Clean Code (40), Head First Design Patterns (38), The Pragmatic Programmer (42), Kotlin in Action (44)
    }

    @Test
    void shouldReturnEmptyListForUnknownTitle() {
        List<Book> books = bookRepo.findByTitle("Unknown Book");
        assertTrue(books.isEmpty());
    }
}
----

Now if you make the same kind of typo we had earlier, the build fails. Try changing the parameter name to:

[source,java]
----
        @Find
        List<Book> findByTitle(String tilte);
----

You should see a compilation error like:

[source]
----
error: no matching field named 'tilte' in entity class 'org.acme.Book'
----

Get the parameter back to the correct name so that we can proceed further.

== Creating new entities

So far we've been reading data, but our bookstore is crowdsourced: users can add books they want to share.
We need to persist new books to the database.

Let's update `Book` to implement `PanacheEntity.Stateless`, which gives us lifecycle operations directly on the entity:

[source,java]
----
@Entity
public class Book extends WithId.AutoLong implements PanacheEntity.Stateless { // <1>
    public String title;
    public String isbn;
    public BigDecimal price;

    public interface Repo extends PanacheRepository.Stateless<Book, Long> { // <2>
        @Find
        List<Book> findByTitle(String title);

        @Query("where price < :maxPrice order by title")
        List<Book> cheaperThan(BigDecimal maxPrice);
    }
}
----
<1> This is the new part: `implements PanacheEntity.Stateless` gives our `Book` class methods to create, delete or update books without the need to use specific methods in repositories.

<2> The repository also changed to `PanacheRepository.Stateless<Book, Long>`.
// TODO: PanacheRepository<Book> infers Long, but Stateless/Managed/Reactive all require <Book, Long> explicitly. We should add single-parameter overloads (defaulting to Long) for consistency.

Now we can write a service that lets users contribute a new book:

[source,java]
----
@ApplicationScoped
public class CrowdSourceService {

    @Transactional // <1>
    public void contributeNewBook(String title, String isbn, BigDecimal price) {
        Book book = new Book();
        book.title = title;
        book.isbn = isbn;
        book.price = price;
        book.insert(); // <2>
    }
}
----
<1> Wraps the method in a database transaction. This is required to define the scope of your operations, so that related operations are committed or rolled back together.
<2> `insert()` immediately executes an `INSERT` statement against the database.

Let's write a test to verify that inserting a new book actually works. Create `src/test/java/org/acme/CrowdSourceServiceTest.java`:

[source,java]
----
package org.acme;

import io.quarkus.test.TestTransaction;
import io.quarkus.test.junit.QuarkusTest;
import jakarta.inject.Inject;
import org.junit.jupiter.api.Test;

import java.util.List;

import static org.junit.jupiter.api.Assertions.*;

@QuarkusTest
public class CrowdSourceServiceTest {

    @Inject
    CrowdSourceService crowdSourceService;

    @Inject
    Book.Repo bookRepo;

    @Test
    @TestTransaction // <1>
    void shouldInsertNewBook() {
        crowdSourceService.contributeNewBook("New Book", "978-1111111111", BigDecimal.valueOf(29.99));
        List<Book> books = bookRepo.findByTitle("New Book");
        assertEquals(1, books.size());
        assertEquals("978-1111111111", books.getFirst().isbn);
        assertEquals(BigDecimal.valueOf(29.99), books.getFirst().price);
    }
}
----
<1> `@TestTransaction` rolls back all database changes when the test method ends. Without it, the inserted book would stay in the database and accumulate on every re-run, causing assertions on counts or prices to fail. Use it on any test that writes data.

[NOTE]
====
If you're an experienced Hibernate user, you'll find this part very different from what you're used to.
When you call `insert()`, `update()`, or `delete()`, an operation is instantly triggered against the database, without calling `.flush()` explicitly.
Managed entities are nowhere to be seen, and neither is the idea of detached and persisted objects.
This is intentional: it's a simpler model that uses `StatelessSession` under the covers.
In this mode every change to an object requires an explicit save operation.
If you're not interested in this style and want the full power of managed entities, skip ahead to <<managed-session>>.
====

=== About entity identifiers

`WithId.AutoLong` is more powerful than Hibernate's traditional `@Id @GeneratedValue` pair: there are variants for different ID types out of the box:

- `WithId.AutoLong`, a generated `Long` id
- `WithId.AutoUUID`, a generated `UUID` id
- `WithId.AutoString`, a generated `String` id

Of course you can always use `@Id` and `@GeneratedValue` directly if you prefer, or if you need a custom ID strategy.
See the xref:quarkus-data-hibernate.adoc#_about_entity_identifiers[reference guide] for the full list of options.

== Managed session

In our bookstore we want to run a promotional sale: apply a discount to all books in a given country, but only if the publisher allows it.
Some publishers offer discounts, others don't.

Our database already has publishers (we set them up in `import.sql`), but we haven't mapped them as entities yet. Let's do that now. Create `Publisher.java`:

[source,java]
----
package org.acme;

import io.quarkus.hibernate.panache.PanacheRepository;
import io.quarkus.hibernate.panache.WithId;
import jakarta.persistence.Entity;

@Entity
public class Publisher extends WithId.AutoLong {
    public String name;
    public String country;
    public boolean allowsDiscounts;

    public interface Repo extends PanacheRepository.Stateless<Publisher, Long> {
    }
}
----

And add a `publisher` relationship to `Book`, plus a stateless repository and a query to find books by publisher country:

[source,java]
----
@Entity
public class Book extends WithId.AutoLong implements PanacheEntity.Stateless {
    public String title;
    public String isbn;
    public BigDecimal price;
    @ManyToOne(fetch = FetchType.LAZY) // <1>
    public Publisher publisher;

    public interface Repo extends PanacheRepository.Stateless<Book, Long> {
        @Find
        List<Book> findByTitle(String title);

        @Query("where price < :maxPrice order by title")
        List<Book> cheaperThan(BigDecimal maxPrice);

        @Query("where publisher.country = :country")
        List<Book> findByPublisherCountry(String country);
    }
}
----
<1> We use `FetchType.LAZY` so that loading a book doesn't automatically join and fetch its publisher.

Now let's implement our discount service:

[source,java]
----
@ApplicationScoped
public class SalesService {

    @Inject
    Book.Repo bookRepo;

    @Inject
    Publisher.Repo publisherRepo;

    @Transactional
    public void applyCountrySale(String country, BigDecimal discountPercent) {
        List<Book> books = bookRepo.findByPublisherCountry(country);
        List<Book> modified = new ArrayList<>(); // <1>
        BigDecimal factor = BigDecimal.ONE.subtract(discountPercent.divide(BigDecimal.valueOf(100)));
        for (Book book : books) {
            Publisher publisher = publisherRepo.findById(book.publisher.id); // <2>
            if (publisher.allowsDiscounts) {
                book.price = book.price.multiply(factor).setScale(2, RoundingMode.HALF_UP);
                modified.add(book); // <3>
            }
        }
        // Other business logic that modifies books i.e. applying loyalty points, rounding prices ...
        // Since we want to update them only once, we collect them to avoid multiple updates within the loops
        for (Book book : modified) {
            bookRepo.update(book); // <4>
        }
    }
}
----
<1> We collect the books we modify so we can update them all at the end, after all business logic has run. Updating inside the body will result in multiple SQL update statements.
<2> In a stateless session, lazy associations are not fetched automatically. We must load the publisher explicitly. A join fetch query in the Book repository could be used instead of a separate fetch, but for the scope of the tutorial we're not introducing join fetches.
<3> We add the book to our list of modified objects.
<4> We update once at the end, after all modifications are done.

Notice how the persistence concerns, tracking which objects changed, fetching lazy associations, explicitly calling `update()`, are mixed in with the business logic. This method is beginning to become complex.

Let's write a test to verify it works:

[source,java]
----
package org.acme;

import io.quarkus.test.TestTransaction;
import io.quarkus.test.junit.QuarkusTest;
import jakarta.inject.Inject;
import org.junit.jupiter.api.Test;

import static org.junit.jupiter.api.Assertions.*;

@QuarkusTest
public class SalesServiceTest {

    @Inject
    SalesService salesService;

    @Inject
    Book.Repo bookRepo;

    @Test
    @TestTransaction // <1>
    void shouldApplySaleOnlyToPublishersThatAllowDiscounts() {
        salesService.applyCountrySale("US", BigDecimal.valueOf(10));

        // Addison-Wesley (US, allows discounts): prices reduced by 10%
        assertEquals(new BigDecimal("40.50"), bookRepo.findByTitle("Effective Java").getFirst().price); // was: 45.00
        assertEquals(new BigDecimal("36.00"), bookRepo.findByTitle("Clean Code").getFirst().price); // was: 40.00

        // Prentice Hall (US, no discounts): prices unchanged
        assertEquals(new BigDecimal("55.00"), bookRepo.findByTitle("Domain-Driven Design").getFirst().price);
        assertEquals(new BigDecimal("50.00"), bookRepo.findByTitle("Refactoring").getFirst().price);
    }
}
----
<1> Rolls back the discounted prices after the test, so other tests and re-runs always see the original data from `import.sql`.

This works, but it can be simpler.
Quarkus Data Hibernate supports a "managed persistence context" that tracks objects for you: Hibernate monitors every entity it loads, detects which fields changed, and generates the minimal SQL needed to keep entities synchronized with the database at commit time. You don't need to track dirty objects, and lazy associations are fetched automatically when you access them.

Switch `Book` to use `PanacheEntity.Managed` and a managed repository:

[source,java]
----
@Entity
public class Book extends WithId.AutoLong implements PanacheEntity.Managed { // <1>
    public String title;
    public String isbn;
    public BigDecimal price;
    @ManyToOne(fetch = FetchType.LAZY)
    public Publisher publisher;

    public interface Repo extends PanacheRepository<Book> { // <2>
        @Find
        List<Book> findByTitle(String title);

        @Query("where price < :maxPrice order by title")
        List<Book> cheaperThan(BigDecimal maxPrice);

        @Query("where publisher.country = :country")
        List<Book> findByPublisherCountry(String country);
    }
}
----
<1> `PanacheEntity.Managed` replaces `PanacheEntity.Stateless`. This tells Hibernate to track this entity in its persistence context.
<2> The repository uses `PanacheRepository<Book>` instead of `PanacheRepository.Stateless`.

Since `Book` is now a managed entity, `insert()` is no longer available. Update `CrowdSourceService` to use `persist()` instead:

[source,java]
----
@ApplicationScoped
public class CrowdSourceService {

    @Transactional
    public void contributeNewBook(String title, String isbn, BigDecimal price) {
        Book book = new Book();
        book.title = title;
        book.isbn = isbn;
        book.price = price;
        book.persist(); // <1>
    }
}
----
<1> `persist()` replaces `insert()`. Unlike `insert()`, it doesn't execute SQL immediately, Hibernate will flush the change when the transaction commits.

Now the service becomes pure business logic:

[source,java]
----
@ApplicationScoped
public class SalesService {

    @Inject
    Book.Repo bookRepo;

    @Transactional
    public void applyCountrySale(String country, BigDecimal discountPercent) {
        BigDecimal factor = BigDecimal.ONE.subtract(discountPercent.divide(BigDecimal.valueOf(100)));
        List<Book> books = bookRepo.findByPublisherCountry(country);
        for (Book book : books) {
            if (book.publisher.allowsDiscounts) { // <1>
                book.price = book.price.multiply(factor).setScale(2, RoundingMode.HALF_UP); // <2>
            }
        }
    } // <3>
}
----
<1> Accessing `book.publisher` triggers a lazy fetch: Hibernate loads the publisher from the database automatically with a query.
<2> We just set the field. No need to call `update()`.
<3> When the `@Transactional` method returns, Hibernate compares every managed entity against its original state and generates the correct SQL. Books whose publisher didn't allow discounts were never changed, so Hibernate skips them entirely.

The test is exactly the same: the behavior hasn't changed, just the implementation got simpler.
It's up to you to decide whether to use a Managed Entity or a Stateless one. The Managed Repository is more expressive, but reasoning about it is more complex and might require knowing some details about how Hibernate handles objects. If you're an experienced Hibernate developer, you probably already know the discussions about n+1 queries, and you expect that changing an object's field doesn't immediately hit the database. Some users prefer a more explicit way to control the database. Either way is fine, and Quarkus Data Hibernate makes it easy to use both and to switch without committing to one style or the other.

== Reactive

If you're already using reactive APIs in your application (Quarkus Messaging, reactive REST clients, Vert.x), you want your database access to compose naturally with them without blocking the event loop.
Quarkus Data Hibernate lets you experiment with both blocking and reactive to see what works best for your case. The same entity model works unchanged with a reactive driver.

First, add the reactive dependencies:

[source,bash]
----
quarkus extension add quarkus-hibernate-reactive
quarkus extension add quarkus-hibernate-reactive-panache
quarkus extension add quarkus-reactive-pg-client
----

NOTE: `quarkus-hibernate-reactive-panache` is needed because `quarkus-hibernate-reactive` does not yet transitively include the classes required by the annotation processor to generate reactive repository implementations (see https://github.com/quarkusio/quarkus/issues/54892[#54892]).

`quarkus-hibernate-reactive` adds the reactive Hibernate engine, which uses non-blocking I/O under the covers instead of JDBC.
`quarkus-reactive-pg-client` is the reactive equivalent of `quarkus-jdbc-postgresql`. It provides a non-blocking PostgreSQL driver built on Vert.x. As its blocking counterpart, it's mandatory for Quarkus Data Hibernate to work.

Now let's change only the minimal part of the code you need to try reactive. Let's add a second nested repository for reactive operations:

[source,java]
----
@Entity
public class Book extends WithId.AutoLong implements PanacheEntity.Managed {
    public String title;
    public String isbn;
    public BigDecimal price;
    @ManyToOne(fetch = FetchType.LAZY)
    public Publisher publisher;

    public interface Repo extends PanacheRepository<Book> { // <1>
        @Find
        List<Book> findByTitle(String title);

        @Query("where price < :maxPrice order by title")
        List<Book> cheaperThan(BigDecimal maxPrice);

        @Query("where publisher.country = :country")
        List<Book> findByPublisherCountry(String country);
    }

    public interface ReactiveRepo extends PanacheRepository.Reactive<Book, Long> { // <2>
        @Find
        Uni<List<Book>> findByTitle(String title); // <3>
    }
}
----
<1> The blocking repository stays unchanged.
<2> A new reactive repository for the same entity. Both repositories can coexist.
<3> The return type is `Uni<T>` instead of `T`. `Uni` is a Mutiny reactive type that represents a value that will be available in the future. Instead of blocking the thread, the method returns immediately and emits the result when the query completes.

To test the reactive repository, you need `quarkus-test-vertx` which provides `@RunOnVertxContext` and `UniAsserter` for testing reactive code. Add it to your `pom.xml`:

[source,xml]
.pom.xml
----
<dependency>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-test-vertx</artifactId>
    <scope>test</scope>
</dependency>
----

Now create `src/test/java/org/acme/ReactiveBookRepositoryTest.java`:

[source,java]
----
package org.acme;

import io.quarkus.test.TestReactiveTransaction;
import io.quarkus.test.junit.QuarkusTest;
import io.quarkus.test.vertx.RunOnVertxContext;
import io.quarkus.test.vertx.UniAsserter;
import jakarta.inject.Inject;
import org.junit.jupiter.api.Test;

import static org.junit.jupiter.api.Assertions.*;

@QuarkusTest
public class ReactiveBookRepositoryTest {

    @Inject
    Book.ReactiveRepo reactiveBookRepo;

    @Test
    @RunOnVertxContext // <1>
    @TestReactiveTransaction // <2>
    void shouldFindBookByTitleReactively(UniAsserter asserter) {
        asserter.assertThat( // <3>
            () -> reactiveBookRepo.findByTitle("Effective Java"),
            books -> {
                assertEquals(1, books.size());
                assertEquals("Effective Java", books.getFirst().title);
            }
        );
    }

    @Test
    @RunOnVertxContext
    @TestReactiveTransaction
    void shouldReturnEmptyListForUnknownTitleReactively(UniAsserter asserter) {
        asserter.assertThat(
            () -> reactiveBookRepo.findByTitle("Unknown Book"),
            books -> assertTrue(books.isEmpty())
        );
    }
}
----
<1> `@RunOnVertxContext` ensures the test runs on the Vert.x event loop, which is required for reactive database access.
<2> `@TestReactiveTransaction` opens a reactive session and transaction for the test, and rolls back at the end.
<3> `UniAsserter` lets you chain reactive assertions without blocking. The lambda passed to `assertThat` returns a `Uni`, and the second lambda verifies the result.

For the full details on reactive sessions, stateless reactive, and combining blocking and reactive code, see the xref:quarkus-data-hibernate.adoc#_let_s_get_reactive[reference guide].

== What you have built

In this guide you went from an empty project to a working database application:

- You started with raw SQL queries using `@SQL` and saw that typos in SQL are only caught at runtime.
- You introduced entities and repositories with compile-time validation via the Hibernate annotation processor.
- You used `PanacheEntity.Stateless` for explicit insert/update/delete operations and saw how persistence logic gets mixed in with business logic.
- You switched to `PanacheEntity.Managed` for automatic dirty tracking and lazy loading, letting you write pure business logic while Hibernate handles the persistence.
- You added a reactive repository that returns `Uni` instead of blocking, using the same entity model.

Every step was verified with tests, so you can be confident the code works.

== Going further

Everything you've seen in this guide is built on top of https://jakarta.ee/specifications/data/[Jakarta Data] and https://jakarta.ee/specifications/persistence/[Jakarta Persistence].
You can use these raw components directly for more advanced scenarios: for instance, you could define repositories as standalone interfaces outside of entities, and the annotation processor will still generate the query implementations for you.
You can also inject the `Session` or `StatelessSession` at any time and do things manually, as you would with the Hibernate ORM extension directly.
The xref:quarkus-data-hibernate.adoc[reference guide] covers the full API surface.
