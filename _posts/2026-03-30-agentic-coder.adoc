It's 2026 and handcrafting code suddenly became a thing of the past. The new trend is vibe coding, and developers are now more like conductors of orchestras of coding agents, that write code, run tests, and fix bugs on their behalf.

Given this premise, we thought that it could have been a good exercise to put at work the capabilities of the LangChain4j agentic framework to build a coding agentic system. Even more interestingly, we went a bit meta and decided to let a coding agentic system, Claude Opus 4.6 in this case, to design and implement the LangChain4j-based coding agentic system, following a vibe coding approach. This article discusses the outcomes of this experiment that produced, as final result, the coding agentic systems, made available in https://github.com/mariofusco/langchain4j-agentic-coder[this project].

== Vibe coding an agentic system

To let Claude to build a first coding agentic system, we simply asked it something like the following:

[source]
----
Study the API and capabilities of the LangChain4j agentic framework from its documentation and source code, and design a coding agentic system based on it that is a clone of yourself.
----

With this prompt, after a few minutes of thinking and working, Claude decided that the https://docs.langchain4j.dev/tutorials/agents#supervisor-design-and-customization[supervisor pattern] was the best fit for the task and came up with an initial architecture for the coding agentic system, like this one:

[source,java]
----
public interface SupervisorCoderSystem {
    @SupervisorAgent(description = """
                    A multi-agent coding assistant that can explore codebases,
                    plan implementations, write/edit code, and run builds/tests.
                    It orchestrates specialized sub-agents to fulfill coding requests.
                    """,
            subAgents = {
                ExplorerAgent.class,
                PlannerAgent.class,
                ImplementerAgent.class,
                ExecutorAgent.class,
        })
    @Override
    String code(@K(UserRequest.class) String request, @K(WorkingDirectory.class) String workingDirectory);

    @SupervisorRequest
    static String request(@K(UserRequest.class) String userRequest, @K(WorkingDirectory.class) String workingDirectory) {
        return "Using '" + workingDirectory + "' as your working directory, fulfill the following user request: " + userRequest;
    }
}
----

Claude also designed and developed, together with their relative system and user messages, the four subagents used by the supervisor, that respectively explore the existing code, create a plan for the actions to take, implement those actions following the plan, and put at work the generated code trying to compile and execute it. Moreover, it also implemented and correctly provided to those agents the necessary tools to perform their tasks, like a file system explorer for the explorer agent, a code editor for the implementer agent, and a way to run code for the executor agent.

This was honestly an impressive result for a first iteration. Moreover, if you have done some vibe coding with Claude before, you know that this is more or less how Claude works in the real world: it generally performs those 4 actions modeled with those 4 agents and orchestrates them in a supervisor-like way to achieve the task at hand. The fact that it was able to design and implement such a system also implies that Claude has some sort of self-awareness of its own internal workings, at least at a high level, and can translate that understanding into a concrete design for an agentic system.

== Putting the agentic coder system at work

To try the coding agentic system designed by Claude, we asked it to write some buggy code and then a test using that system to fix it. Claude generated the following `Calculator` class with 4 slightly bugged methods

[source,java]
----
/**
 * A simple calculator that performs basic arithmetic operations on lists of numbers.
 */
public class Calculator {

    /**
     * Returns the sum of all numbers in the list.
     */
    public int sum(List<Integer> numbers) {
        int total = 0;
        for (int i = 0; i <= numbers.size(); i++) {
            total += numbers.get(i);
        }
        return total;
    }

    /**
     * Returns the average of all numbers in the list.
     */
    public double average(List<Integer> numbers) {
        if (numbers.isEmpty()) {
            return 0;
        }
        return sum(numbers) / numbers.size();
    }

    /**
     * Returns the maximum value in the list.
     * Throws IllegalArgumentException if the list is empty.
     */
    public int max(List<Integer> numbers) {
        if (numbers.isEmpty()) {
            throw new IllegalArgumentException("List must not be empty");
        }
        int max = 0;
        for (int n : numbers) {
            if (n > max) {
                max = n;
            }
        }
        return max;
    }

    /**
     * Returns the factorial of n.
     * Throws IllegalArgumentException if n is negative.
     */
    public long factorial(int n) {
        if (n < 0) {
            throw new IllegalArgumentException("n must be non-negative");
        }
        long result = 1;
        for (int i = 1; i < n; i++) {
            result *= i;
        }
        return result;
    }
}

[source,java]
----
public class Calculator {

    public int sum(List<Integer> numbers) {
        int total = 0;
        // Use for-each to avoid index errors and handle empty lists correctly
        for (int n : numbers) {
            total += n;
        }
        return total;
    }

    public double average(List<Integer> numbers) {
        if (numbers.isEmpty()) {
            return 0;
        }
        // Ensure floating-point division to return fractional averages
        return (double) sum(numbers) / numbers.size();
    }

    public int max(List<Integer> numbers) {
        if (numbers.isEmpty()) {
            throw new IllegalArgumentException("List must not be empty");
        }
        // Initialize to the first element to correctly handle all-negative lists
        int max = numbers.get(0);
        for (int i = 1; i < numbers.size(); i++) {
            int n = numbers.get(i);
            if (n > max) {
                max = n;
            }
        }
        return max;
    }

    public long factorial(int n) {
        if (n < 0) {
            throw new IllegalArgumentException("n must be non-negative");
        }
        long result = 1;
        // Include n in the multiplication to compute n!
        for (int i = 1; i <= n; i++) {
            result *= i;
        }
        return result;
    }
}

== From supervisor to workflow

After this first experiment, we asked Claude to redesign the coding agentic system using a workflow-based approach instead of a supervisor-based one, and it came up with the following architecture, where the workflow is a sequence made by five steps that are similar to the four former ones, plus a fifth that prints the summary of all the actions taken during the execution, something that the supervisor does autonomously without the need of a specialized agent.

[source,java]
----
public interface WorkflowCoderSystem extends MonitoredAgent {

    @SequenceAgent(
            description = "A workflow-based coding pipeline: explore, plan (with review loop), "
                    + "implement, execute (with evaluation and refactoring loop), then summarize",
            typedOutputKey = Summary.class,
            subAgents = {
                ExplorerAgent.class,
                PlanReviewLoop.class,
                ImplementerAgent.class,
                ExecutionLoop.class,
                SummarizerAgent.class
            })
    @Override
    String code(@K(UserRequest.class) String request, @K(WorkingDirectory.class) String workingDirectory);
}
----

The other major difference with the supervisor-based implementation is that, this time both the planning and the execution steps are not single agents, but are implemented with a loop structure, that allows those steps to perform multiple iterations and eventually adjust and improve their work before moving forward.

For example the execution loop is made by three sub-agents, an executor, an evaluator, and a refactor agent, that are iteratively invoked until the evaluation score is good enough (greater than or equal to 0.8 in this case) or the maximum number of iterations is reached.

[source,java]
----
public interface ExecutionLoop {

    @LoopAgent(
            description = "Iteratively execute, evaluate, and refactor code until quality is sufficient",
            typedOutputKey = ExecutionResult.class,
            maxIterations = 5,
            subAgents = {ExecutorAgent.class, EvaluatorAgent.class, RefactorAgent.class})
    String executeAndRefine(
            @K(ImplementationResult.class) String implementationResult,
            @K(WorkingDirectory.class) String workingDir);

    @ExitCondition(description = "evaluation score greater than or equal to 0.8")
    static boolean exit(@K(EvaluationScore.class) double score) {
        return score >= 0.8;
    }
}
----

This time however, despite the higher number of agents involved, the complete debugging session is performed in only two minutes, against the more than six necessary for the supervisor-based version, because of the overhead introduced by the supervisor agent itself, that has to coordinate all the other agents, autonomously generating their invocation and the appropriate arguments to invoke them.

== Conclusions

In this article we have seen how to use the LangChain4j agentic framework to build a coding agentic system following a vibe coding approach, letting an LLM to design and implement the system itself. In this context, the API of the LangChain4j agentic framework proved to be straightforward and easy to use enough to allow the LLM to autonomously design and implement a quite complex agentic system, and at the same time powerful enough to allow Claude to clone its own internal workings in that system, creating a sort of meta-coding agentic system.

We have also seen how to put that system at work on a real debugging session, and how to monitor its execution and visualize its topology. The results were quite impressive, with the system being able to fix all the bugs in the code and make all the tests pass, and with a quite articulated topology of agents working together to accomplish the task. Moreover, we have seen how a workflow-based approach can be more efficient than a supervisor-based one, even if it involves more agents, because of the lower overhead.
